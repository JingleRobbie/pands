# Work Order PDF Parser — Claude Code Brief

## Project Context

A SvelteKit web app is being built to replace an existing Excel-based work order system.
One requirement is the ability to upload a PDF work order and have it automatically parsed
into structured data, eliminating manual data entry. This brief covers the parser module only.

## Stack

- SvelteKit (no TypeScript)
- MySQL + mysql2 (promise API)
- Tailwind CSS v3
- Plain JavaScript throughout

---

## PDF Characteristics

- Generated by the same software every time — consistent template, stable layout
- Fixed column positions across all documents
- Multi-page documents are supported; page 1 contains the header and begins the line items;
  pages 2+ contain only continuation line items (no header repeat)
- Line item row numbers are sequential across all pages — no renumbering needed
- The existing Excel template displays 18 line item slots per page as a display artifact
  of that format. The actual number of line items on any given work order varies — a work
  order may have any number of rows. The parser captures every row that contains data
  across all pages, with no assumed limit.

---

## Data to Extract

### 1. Header Fields (page 1 only)

The header spans three logical lines. Each line must be parsed by splitting at known
label boundaries in order — do not search for labels independently across the full page.

**Header line 1** — single line, four fields:

| Field | Type | Terminated by |
|---|---|---|
| `customer` | Text | `Branch:` |
| `branch` | Text | `Date Rec'd:` |
| `date_received` | Date string as printed e.g. "4/2/2026" | `Deliver On:` |
| `deliver_on` | Date string as printed e.g. "4/9/2026" | end of line |

**Header line 2** — single line, three fields:

| Field | Type | Terminated by | Notes |
|---|---|---|---|
| `contact` | Text | `Phone:` | |
| `phone` | Text | `PO #:` | Strip all non-numeric characters |
| `po_number` | Text | end of line | May be blank |

**Header line 3** — job info and address area:

| Field | Type | Notes |
|---|---|---|
| `job_name` | Text | Always on the same line as the `Job Name:` label |
| `delivery_address_1` | Text | First line of address e.g. "1149 NE QUAIL CREEK RD" |
| `delivery_address_2` | Text | Second line of address e.g. "LAWTON, OK 73507" |
| `tax_rate` | Text | Percentage string e.g. "8.5%" — may be blank |

### 2. Line Item Rows (all pages, flat array)

Each row that contains data should be captured. Empty rows are skipped.

| Field | Type | Notes |
|---|---|---|
| `row` | Integer | Sequential row number as printed |
| `facing` | Text | e.g. "VRR+", "Black", "Unfaced", "Other" — treat as plain text |
| `roll_qty` | Integer | Whole number, less than 999 |
| `thickness` | Float | One decimal place only if present e.g. 2.5, 3, 3.5, 4, 6, 8, 9.5 |
| `width` | Integer | e.g. 36, 48, 60, 72 — treat as plain integer |
| `length` | Float | Max 200, decimal if present is always .5 e.g. 86.5 |
| `roll_for` | Text | e.g. "Roof", "SW", "Sidewall", "EW", "Endwall", "Stock" — treat as plain text |
| `instructions` | Text | Free text, varying length, may be blank. Font may be small in PDF due to Excel shrink-to-fit — pdfjs-dist will extract it regardless of size |
| `tab` | Text | Derived from facing cell background color — see TAB Color Detection note below |

### 3. MBI Accessories (page 1 only, fixed slots)

Parse only the filled slots. Empty slots are skipped.

The parser must locate the MBI Accessories section by its label anchor and extract
only the rows within that section — it must not pick up numbers from the totals area
or any other nearby section.

Each filled slot contains:

| Field | Type | Notes |
|---|---|---|
| `qty` | Integer | |
| `part_number` | Text | Split the part string at the first space — everything before the first space is the part number. Alphanumeric, may contain dashes. Discard everything after the first space. |

Example correct output:
```json
"accessories": [
  { "qty": 10, "part_number": "58952" },
  { "qty": 1,  "part_number": "58970" }
]
```

## Fields to Ignore

- Summary-Faced table
- Summary-Unfaced table
- SqFt/Roll, Total SF, Cubic Feet columns (calculated by the app at view/print time)
- Footer totals (Faced Rolls, Total Cubic Feet, Total SF, Total Lineal Ft)
- "SELL..." pricing notes
- "NO DEL. CHG." note
- TAB checkbox labels and totals area (the per-row TAB value is derived from color, not the checkboxes)
- Complete Date, Lead, Support 1, Support 2 columns

---

## Parsing Approach

Use `pdfjs-dist` running client-side in the browser (no server-side parsing).

The library returns a flat array of text items, each with an `x` and `y` coordinate.
The parsing strategy is:

1. **Group items by Y position** with a small tolerance (~2px) to reconstruct logical rows
2. **Identify sections** by known anchor text (e.g. "ROW", "MBI Accessories", "Customer:")
3. **Map X ranges to column names** — because the template never changes, column X boundaries
   can be defined as constants after inspecting one sample document. These constants will be
   stable for all future documents.
4. **Page detection** — detect whether a page has a header by checking for the presence of
   "Customer:" anchor text. If absent, treat the page as a continuation page and parse
   line items only.
5. **Header parsing** — parse each header line by splitting the full line string at known
   label boundaries in sequence, not by searching for labels independently.

## Output Structure

The parser should return a single plain JavaScript object:

```js
{
  header: {
    customer: "",
    contact: "",
    phone: "",
    branch: "",
    date_received: "",
    deliver_on: "",
    po_number: "",
    job_name: "",
    delivery_address_1: "",
    delivery_address_2: "",
    tax_rate: ""
  },
  line_items: [
    {
      row: 1,
      facing: "",
      roll_qty: 0,
      thickness: 0,
      width: 0,
      length: 0,
      roll_for: "",
      instructions: "",
      tab: ""
    }
    // ...
  ],
  accessories: [
    {
      qty: 0,
      part_number: ""
    }
    // ...
  ]
}
```

## Deliverables

1. **`src/lib/parsers/parseWorkOrder.js`**
   - A single exported async function: `export async function parseWorkOrder(file)`
   - Accepts a browser `File` object (from a file input)
   - Returns the structured object described above
   - Uses `pdfjs-dist` for text extraction
   - Self-contained — no SvelteKit-specific imports, so it can be tested in isolation

2. **`src/routes/upload/+page.svelte`** (minimal)
   - A file input that accepts PDF only
   - On file selection, calls `parseWorkOrder(file)` and logs the result to the console
   - No styling required — this is a test harness, not production UI

3. **A brief note** in comments within the parser identifying where the X-boundary
   constants are defined, so they are easy to find and adjust if the template ever changes.

## Notes for Claude Code

- Do not use TypeScript
- Do not use any server-side PDF processing — everything runs in the browser
- Install `pdfjs-dist` via npm
- The PDF worker should be configured appropriately for SvelteKit's Vite build
- If `pdfjs-dist` worker configuration in SvelteKit/Vite requires a specific setup, handle it
- Do not build any database layer — this brief covers parsing only
- The output object feeds into a SvelteKit form action in a separate task

## TAB Color Detection — Open Question

Each line item row has a `tab` field whose value is determined by the background color
of that row's facing column cell in the PDF:

- White background → `"1x6"`
- Yellow background → `"2x3"`
- Black background → `"2x9"`

**This is an unresolved technical question.** It is not yet confirmed whether `pdfjs-dist`
exposes cell background fill colors in its output. Before implementing the `tab` field:

1. Run a diagnostic pass on a real work order PDF using `pdfjs-dist`'s operator list
   (`page.getOperatorList()`) for a page containing colored rows
2. Check whether fill color operators (e.g. `setFillRGBColor`) are present and can be
   correlated to the Y position of each line item row
3. If `pdfjs-dist` operator list exposes the colors, implement the mapping from RGB
   values to TAB strings. The exact RGB values for white, yellow, and black should be
   logged from the diagnostic pass rather than assumed.
4. If `pdfjs-dist` cannot expose the colors, flag this to the developer — an alternative
   library such as `pdf-lib` or raw PDF stream parsing may be required for this field only.
   The rest of the parser should still function and return `tab: ""` as a placeholder.
