## Shipped / Completed

- **SO edit page** (`/so/[id]/edit`) — full implementation of edit flow: header fields
  (SO number, customer, job name, ship date) + line items. Lines with production runs
  are shown read-only ("locked"), lines without runs are editable. Existing lines can be
  deleted (removed from `keptIds`) or have sqft/facing updated. New lines can be added.
  Server-side guards: status check (OPEN/IN_PROGRESS only), duplicate SO number check,
  sqft-cannot-go-below-produced check.

- **Facing field on new SO** (`/so/new`) — `facing` column added to new SO line items
  (Faced / Raw dropdown). Server action now reads `facings` array and stores it. Schema
  updated to make `facing` non-null with default `'Faced'`.

- **SO-level confirmation** (`/so/[id]/confirm`) — new page that shows all scheduled
  runs for a job in one table, each with an editable "Actual Sqft" input. Confirms all
  runs at once. Already-completed runs shown in a secondary read-only table. Matrix
  outlook embedded below.

- **Production list "Confirm" button** (`/production`) — Confirm buttons on today's runs
  now navigate to `/so/[so_id]/confirm` (SO-level confirm) instead of the per-run confirm
  page. The per-run confirm page still exists for direct access.

- **History range selector on matrix** (`/matrix`) — dropdown to show historical rows:
  current (hidden), last 7d, 30d, or 365d. `getMatrixData()` in `inventory.js` always
  fetches 365d of history; the UI filters client-side by the selected range.

- **Schema change** — `facing` column on `sales_order_lines` altered to `NOT NULL DEFAULT 'Faced'`.

## Decisions Made

- SO-level confirmation is the primary confirm workflow (one form, all runs for a job),
  accessible via Confirm button on today's production runs. The per-run `/production/[id]/confirm`
  page is preserved but demoted to secondary access.

- History window is fixed at 365 days server-side and filtered client-side. This keeps
  the query simple and avoids a server round-trip on range change.

- Lines with any production runs attached are fully locked on edit (shown read-only). The
  guard is `run_count > 0` from a subquery in the load, not a status check.

## Bugs & Issues

- **Fixed:** `facing` not being saved on new SO creation — the `facing` column was missing
  from the `INSERT INTO sales_order_lines` in `/so/new/+page.server.js`.

- **Open:** `subtractDays` in `inventory.js:373` is called with `365` but the function
  _subtracts_ days, so it computes a date 365 days in the past correctly — however it's
  also called as `subtractDays(fromDate, 1)` in `getBalancesAsOf` which means baseline
  is 1 day before `fromDate`. This appears intentional but worth re-checking if history
  rows drift.
