## Shipped / Completed

- **Partial confirm support:** `confirmRun` in `src/lib/services/production.js` now accepts actual
  sqft that can be less than scheduled. Validates actual ≤ scheduled (throws if over). Auto-creates
  a new UNSCHEDULED run for the shortfall sqft, returning `{ shortfallRunNumber, shortfallSqft }`.
- **Run date override on confirm:** `confirmRun` accepts an optional `runDate` param; uses
  `COALESCE(?, run_date)` so the date is updated only when provided.
- **Delete run:** New `deleteRun` service function — deletes any non-COMPLETED run in a transaction.
- **Confirm page overhaul** (`src/routes/so/[id]/confirm/`):
    - Default form action split into named `confirm` and `remove` actions.
    - Date input per run (pre-filled, editable before submitting).
    - Delete button (×) per run with a native `<dialog>` confirmation modal.
    - After confirm, page shows success state with shortfall run numbers instead of redirecting.
    - `max` attribute on sqft input enforces actual ≤ scheduled in the browser too.
- **Calendar status filter** (`src/routes/calendar/`):
    - Filter tabs (All / Scheduled / Completed) on the calendar page.
    - `status` query param wired through to `/calendar/events` server endpoint with SQL branching.

## Decisions Made

- Partial confirms auto-create a shortfall run rather than requiring manual re-scheduling — keeps
  the SO line fully accounted for without extra steps.
- Confirm page stays on the same page after success (shows summary) rather than redirecting to
  `/production` — lets the user see shortfall run numbers before navigating away.
- Run date is editable at confirm time (not locked to scheduled date) — allows recording the actual
  production date if it differed.
- `deleteRun` restricted to non-COMPLETED runs only — completed runs are immutable (have inventory
  transactions).
