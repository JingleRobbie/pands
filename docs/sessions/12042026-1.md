## Shipped / Completed

- **PO receiving rework (full):** Removed the manual receive step entirely. Balance is now
  derived from PO lines where `expected_date <= CURDATE()` (dual-source: PO lines +
  inventory_transactions). Deleted `src/lib/services/purchasing.js` and the entire
  `src/routes/po/[id]/receive/` route. PO detail page is now read-only with a Cancel
  button (`use:enhance` confirm guard). PO list drops the Overdue section and Receive
  buttons. Matrix history rows show muted "received" text instead of a green badge.

- **Header normalization:** All four PO list views (Active, Open, Cancelled, All) now share
  the same columns: PO #, Vendor, Expected, Status, Lines. SO lists were already consistent.
  Production lists (Overdue, Today, Upcoming, Search) standardized to include Status and
  SO # everywhere.

- **Clickable rows:** `onclick={() => goto(href)}` on `<tr>` elements across PO, SO, and
  production list pages. Action cells (Edit, Confirm buttons) use `e.stopPropagation()`.

- **SO-level confirm page (`/so/[id]/confirm`):** New route. Clicking Confirm on any
  production run navigates to this page showing all SCHEDULED runs for that SO, each with
  an editable actual sqft input defaulting to scheduled sqft. Single submit calls
  `confirmRun()` per run in sequence. Already-completed runs shown as greyed context below.

- **CONFIRMED → COMPLETED rename:** Production run status value renamed throughout — 9
  source files updated, DB ENUM altered via 3-step migration (expand → UPDATE 12 rows →
  shrink), nav tab updated, `statusBadge` mapping updated.

- **Production list queries:** Added `so.id AS so_id` to all four queries so Confirm buttons
  can link to `/so/${run.so_id}/confirm`.

## Decisions Made

- **Option A for SO confirm:** New `/so/[id]/confirm` route (SO-scoped) rather than
  expanding the existing run-level page. Reasoning: the conceptual unit being confirmed
  is the job (SO), not an individual run.

- **PO "Upcoming" filter:** Shows `status != 'CANCELLED'` (not just OPEN), so any non-
  cancelled future PO appears regardless of status edge cases.

- **`confirmed_at` / `confirmed_by` DB columns left as-is:** Renaming those columns would
  require a schema migration and offer no functional benefit. Tolerable mismatch.

## Bugs & Issues

- **Fixed:** `/so/9/confirm` appeared to show "no runs to confirm" — root cause was the
  `TODO(human)` placeholder in the `{#each scheduledRuns}` block left the table body
  empty. Data query was correct all along (SO 9 had 2 SCHEDULED runs). Fixed by
  implementing the table row (hidden `run_id` input + number input for actual sqft).

- **Fixed:** Hidden `<input type="hidden">` was placed directly inside `<tr>` outside any
  `<td>` — browsers silently hoist it out of the table, which can break form submission.
  Moved it inside the last `<td>` alongside the sqft input.

- **Fixed:** CONFIRMED→COMPLETED DB migration failed on first attempt because `status` is
  an ENUM. Required 3-step: add COMPLETED to ENUM, UPDATE rows, remove CONFIRMED from ENUM.

- **Open:** The single-run confirm page (`/production/[id]/confirm`) still exists and still
  works but is no longer the primary confirm path. Row clicks on the production list still
  navigate there for viewing a run's details.
