# Session — 13 May 2026, Session 1

**Duration:** ~1 hour
**Branch:** main

## Shipped / Completed

- **Bug fix:** `wo/[id]/schedule` was showing ALL work order lines (including CUT_SHIP,
  DIRECT_SHIP, and BILLING lines) when it should only show lines that require laminate
  production (STANDARD + CUT_LAMINATE). Fixed by adding `child_count` subquery to the
  load query and filtering via `inferPathType()` from `line-paths.js`.

- **CUT_LAMINATE grouping:** Billing parent lines now render as group header rows on the
  schedule page. Their child CUT_LAMINATE lines appear indented beneath, matching the
  visual grouping pattern established in `wo/[id]`.

- **Child deduplication:** Same-spec children under a billing parent (same rollfor, facing,
  thickness, width, length) are collapsed into one input showing the combined
  `rollsUnscheduled`. On submit, `distributeGroupRolls()` fills children sequentially (child 1
  up to its max, remainder to child 2). Hidden `<tr>` rows emit individual `line_id`/`rolls`
  pairs so the action's zip logic is unchanged.

- **Cut-Down Pending badge:** Added `confirmed_cut_down_count` subquery to the schedule load.
  Billing group header shows an amber `badge-amber "Cut-Down Pending"` when the count is 0.

## Decisions Made

- Sequential distribution (not proportional) for grouped child roll input — simpler and
  predictable; fills the first child to capacity before allocating to the next.
- Cut-down pending is a **warning only** — scheduling is not blocked. User can still create
  production runs before cut-down is confirmed.
- Action code in `+page.server.js` left unchanged — hidden rows maintain the 1:1
  `lineIds[i] / rolls[i]` zip contract the action relies on.
